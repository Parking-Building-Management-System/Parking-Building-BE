# 07. Owner Recommended Decisions

| Decision | Recommended Choice | Reason | Trade-off | Affected Modules |
|---|---|---|---|---|
| tenant isolation strategy | Keep Hibernate Filter for current phase, add global request-level `TenantContext` binding for non-admin routes, and explicit admin bypass. | Code already has `TenantScopedEntity`, `TenantFilterAspect`, `TenantContext`; smallest safe foundation. | Requires careful tests; native queries still need review. | identity, admin, manager, future staff/PWA |
| tenant_id placement | Direct `tenant_id` on every core business table; keep identity `sessions` indirect via `user` unless owner requires direct session tenant_id. | Most current business tables already follow this; direct FK simplifies isolation and indexes. | Direct tenant_id duplicates some relation data. | parking, operation, billing, audit, notification |
| role model | Keep global roles/permissions with tenant-scoped users. | Current schema and JWT authority cache already work this way. | Tenant-specific custom roles need later extension. | identity, admin, security |
| staff login identifier | Use internal username, globally unique as `<tenantSlugOrCode>.<staffCode>` for MVP. | Avoids personal email dependency while fitting current global `users.username` unique constraint. | Username includes tenant prefix; owner must define code convention. | identity, manager staff |
| manager login identifier | Keep manager email for System Admin provisioning in MVP. | Current `AdminTenantProvisionRequest` and service already use manager email. | If managers need non-email usernames later, provisioning DTO changes. | admin tenant provisioning, identity |
| vehicle type global vs tenant-specific | Keep VehicleType global; make pricing tenant-specific. | Current `vehicle_types` is global and admin CRUD exists. Tenant-specific pricing covers business variability. | Some tenants may want custom type names; can add display override later. | master data, facility, pricing |
| parking ownership | Let System Admin provision tenant only; let Manager create/manage parkings unless owner wants admin-controlled buildings. | B2B tenant managers know their facility layout; current manager APIs already manage structure below parking. | Requires adding manager parking CRUD and strong tenant isolation. | admin, manager facility |
| parking/floor/zone/slot hierarchy | Keep `Parking -> Floor -> Zone -> Slot`; Zone has vehicle type; Slot has floor and zone. | Matches current schema and manager topology API. | Older `zones.floor_name` is legacy compatibility; prefer `floor_id`. | parking, manager facility |
| pricing phase timing | Defer pricing to PHASE_3_BILLING after entry/exit basics. | Pricing depends on session lifecycle and payment model. | Staff cash flow initially cannot compute real bills. | operation, billing |
| subscription phase timing | Defer full subscription CRUD/jobs to PHASE_3_BILLING. | Entity exists but no service/API/job. | PWA monthly pass waits. | subscription, billing, PWA |
| payment mock vs real integration | Start with mock payment/VietQR/webhook; defer real gateway. | Spec says phase can simulate; faster and safer for workflow validation. | Real reconciliation/integration later. | billing, PWA checkout |
| device binding MVP scope | Use existing `devices` with `PENDING/APPROVED/REJECTED/SUSPENDED`, `expires_at` for temporary approval; add `DeviceApprovalRequest` only if audit history is required now. | Minimal schema change and matches current login flow. | Less historical detail if no separate request table. | identity, manager device |
| blind drop phase timing | Do not include in CRUD phase; implement after StaffShift and Payment ledger exist. | Blind drop needs expected cash from completed sessions/payments. | Shift handover screen waits. | operation, billing, audit |
| red flag audit MVP scope | Add `RedFlagAction` as tenant-scoped workflow entity and write `AuditLog` for each action. | Generic `AuditLog` alone cannot enforce reason/evidence/session requirements. | New table/API in operations phase. | operation, audit |
| PWA lazy auth phase timing | Defer to PHASE_4_PWA; first decide driver identity model. | Phone+plate OTP has security implications and may not reuse current staff/manager auth cleanly. | PWA quick access waits. | identity, PWA |
